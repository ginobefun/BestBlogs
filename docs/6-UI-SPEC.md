## 1. 文档定位

本文件用于承接 `5-DESIGN.md` 中定义的设计原则，并将其转化为可执行的 UI 规范。
如果说 `5-DESIGN.md` 是 BestBlogs 的体验操作系统，那么 `6-UI-SPEC.md` 就是体验原则的执行手册。

**本文档主要回答：**

1. 颜色、字体、间距、布局如何使用；
2. 组件与状态如何落地；
3. 内容对象如何在列表、卡片和详情页中呈现；
4. 不同页面类型如何组织 UI；
5. 扫描态、理解态、深读态、回顾态、选择态如何落地；
6. AI 伴读、沉浸式翻译和 Pro 引导如何克制出现；
7. 前端与设计执行时应遵循哪些统一规则。

本文档不负责重新定义品牌定位、产品结构和体验原则。  
若具体实现与顶层设计原则冲突，以 `5-DESIGN.md` 为准。  
若视觉表达与品牌边界冲突，以 `3-BRAND.md` 为准。  
若文案命名与术语冲突，以 `10-TERMINOLOGY.md` 为准。

## 2. 总体执行原则

### 2.1 语义优先于硬编码

优先使用设计 token 和语义变量，不直接在新代码中使用硬编码颜色和零散样式。

**执行要求：**

- 新代码优先使用语义 token；
- 不继续扩散 `slate-*`、`gray-*` 等零散硬编码色；
- 不在组件内部随意定义一次性颜色；
- 新增语义 token 需要同步 light / dark；
- token 名称应表达用途，而不是表达颜色本身。

**推荐：**

```css
color: var(--text-primary);
background: var(--bg-surface);
border-color: var(--border-muted);
```

**避免：**

```tsx
className="text-slate-700 bg-orange-50 border-gray-200"
```

### 2.2 复用优先于新增

同类对象尽量复用已有组件与样式，不轻易另起一套视觉语言。

**执行要求：**

- 同类卡片优先复用统一结构；
- 同类 badge / tag 使用统一组件；
- 同类空状态使用统一模式；
- 新增组件前先确认是否能通过 variant 承接；
- 不为单个页面另起一套视觉系统。

### 2.3 内容对象优先于装饰元素

卡片、列表项、详情页的重点是标题、摘要、来源、标签、状态，而不是视觉填充。

**执行要求：**

- 标题优先于封面；
- 摘要优先于装饰；
- 来源和时间优先于泛化图标；
- 不为无图内容强行补无意义占位图；
- 不用大图、大色块、大阴影制造虚假重要性。

### 2.4 响应式不是等比缩放

移动端与桌面端要根据任务差异调整信息优先级，而不是简单压缩桌面布局。

**执行要求：**

- 移动端优先快速判断和轻量操作；
- 桌面端可以承载更多辅助信息；
- 不允许关键判断信息只在 hover 出现；
- 中小屏不强制保留桌面三栏结构；
- 阅读页移动端优先正文体验。

### 2.5 克制优先于显眼

BestBlogs 的 UI 不以抢眼为目标。

**执行要求：**

- CTA 应清楚，但不制造焦虑；
- Pro 引导应上下文相关，而不是强打断；
- AI 入口应可见，但不抢正文主角；
- Badge 和信号应帮助判断，而不是堆叠存在感；
- 动效服务理解，不服务炫技。

## 3. 设计 Token

### 3.1 品牌色

BestBlogs 的核心视觉母题是 Ink + Amber + Paper。

**当前基础色：**

| 名称 | 色值 | 语义 |
| --- | --- | --- |
| Ink | #1a365d | 知识深度、专业判断、稳定可信 |
| Amber | #d97706 | 经过判断后值得注意的高亮 |
| Paper | #fefdfb | 承托阅读的温和背景与安静表面 |

这三个颜色不是装饰色，而是体验语义。

### 3.2 语义变量

**变量来源：**

- bestblogs-app/src/styles/globals.css
- bestblogs-app/tailwind.config.ts

**建议维持以下主语义层：**

#### 背景

- --bg-paper
- --bg-surface
- --bg-muted
- --bg-elevated

#### 文本

- --text-primary
- --text-secondary
- --text-tertiary
- --text-muted
- --text-inverse

#### 边框

- --border-default
- --border-muted
- --border-emphasis

#### 交互

- --color-link
- --color-link-hover
- --focus-ring
- --selection-bg

#### 状态

- --color-success
- --color-warning
- --color-error
- --color-info

### 3.3 Amber 使用规范

Amber 仅用于「值得注意」的信号位，不用于制造热闹感。

**推荐用法：**

| 场景 | 建议 |
| --- | --- |
| 侧边栏 active 项 | 弱 amber 背景 + Ink 文本 |
| Featured / 精选内容 | 左侧强调线或小面积高亮 |
| 主 CTA 按钮 | Amber 主按钮 |
| 评分 / 星标 | Amber 文本或图标 |
| Pro 标识 | 弱 amber 底 + 深色文字 |
| 选中态指示器 | 细窄 amber 指示线 |
| 推荐原因强调 | 轻量 amber icon 或文本强调 |

**禁止：**

- 破坏性操作；
- 普通正文文字；
- 大面积背景填充；
- 纯装饰边框；
- 高频闪烁；
- 让整个页面呈现促销感。

### 3.4 暗色模式

**暗色模式要求：**

- 使用 next-themes + class 切换；
- 新增 light token 时必须补齐 dark token；
- 暗色背景使用 near-black，而非纯黑；
- 暗色文字使用 off-white，而非纯白；
- 暗色下卡片阴影应转为更克制的边框与轻阴影组合；
- Amber 在暗色下可以适当亮化，但不提高饱和度到抢眼程度。

**执行要求：**

- 不允许只在浅色模式下验证页面；
- 所有新增组件必须验证 light / dark；
- focus ring 在浅色和暗色下都必须清晰；
- 状态色在暗色下必须满足可读性；
- 深读页暗色模式避免高对比刺眼。

## 4. 字体与排版

### 4.1 字体角色

**当前字体语义：**

| 角色 | 用途 |
| --- | --- |
| font-heading | 标题、模块标题、内容标题 |
| font-body / font-sans | 正文、UI、说明文案 |
| font-mono | 代码、技术字段 |
| font-numeric | 数值、统计、金额、指标 |

**约束：**

- 全站标题统一使用 font-heading；
- 正文和 UI 不混用装饰性 serif；
- 代码区只使用等宽字体；
- 数值型信息优先使用数字字体以保证对齐；
- 不为单页营销效果单独引入额外字体风格。

### 4.2 排版层级

**核心三层语义模型：**

| 层级 | 用途 | 建议 |
| --- | --- | --- |
| L1 | 卡片 / 列表项标题 | text-lg font-heading font-semibold leading-snug |
| L2 | 摘要 / 描述 | text-sm text-muted-foreground leading-relaxed |
| L3 | 来源 / 日期 / 时长 | text-xs text-muted-foreground |

**补充规则：**

- L1 与 L2 字号差距至少 4px；
- L3 必须明显弱于 L2；
- 页面 H1 使用 text-2xl md:text-3xl font-heading font-bold；
- 模块标题不要过度放大；
- 内容列表中标题优先于缩略图和操作按钮。

### 4.3 编辑说明与推荐理由

**BestBlogs 中有大量编辑式说明，例如：**

- 编者按；
- 推荐理由；
- 今日重点；
- 为何精选；
- 为什么推这条；
- 回顾说明；
- 主题解读摘要。

**执行要求：**

- 编辑说明应区别于正文，但不应像广告文案；
- 推荐理由应轻量、可信、可解释；
- 不使用夸张形容词；
- 不使用感叹号制造情绪；
- 不用大字号和强色块抢正文注意力。

**建议样式：**

- text-sm
- leading-relaxed
- text-secondary
- 轻微背景或左侧弱强调线
- 不超过必要长度

### 4.4 元信息格式

**统一格式：**

来源名 · 日期 · 阅读时长

**补充要求：**

- 分隔符统一使用 ·；
- 最多 3 个维度；
- 评分 / 星标独立展示，不混入元信息行；
- 元信息默认单行；
- 优先级顺序：来源 > 日期 > 成本 / 时长；
- 不把营销语和功能性文字混入元信息区；
- 移动端可折行，但不能丢失来源。

### 4.5 标签规则

统一使用 BbBadge 或等价统一组件。

**执行要求：**

- 每卡最多 3 个标签；
- 多余折叠为 +N；
- 标签只起补充判断作用；
- 不得用标签替代标题与摘要；
- 分类 / 语言 / 主题 / 状态标签应有稳定 variant；
- 标签行不应压缩标题可读性；
- 不形成 badge 墙。

**常见 variant：**

- category
- language
- topic
- status
- pro
- featured
- source
- personalized

### 4.6 正文排版

**详情页正文区执行要求：**

- 控制单行长度；
- 行高优先阅读舒适度；
- 标题、正文、引用、列表、代码块清晰分层；
- 正文中的强调色极度克制；
- 正文区不嵌入与阅读无关的高噪音操作；
- 长文阅读中工具条不应持续抢占视觉中心；
- 双语对照时原文与译文关系清楚，不破坏连续阅读。

## 5. 间距系统

### 5.1 语义间距

采用 4px grid。

| Token | 值 | 用途 |
| --- | --- | --- |
| tight | 8px | 同一元素内部间距 |
| element | 16px | 相关元素之间 |
| component | 32px | 独立组件之间 |
| section | 80px | 页面大区块，主要用于品牌说明页 |

### 5.2 间距使用规则

**推荐：**

- 页面标题区 → 内容区：mb-component
- 筛选器 → 列表：mb-component
- 列表项之间：space-y-element 或 gap-element
- 卡片内部：p-element，内部 gap-tight
- 侧边栏分组：组间 mt-component
- 标题上下比例：上方间距大于下方

**执行补充：**

- 卡片内部偏紧，组件之间偏松；
- 品牌说明页 section 间距可以更大；
- 设置页不要沿用营销页的大节奏；
- 阅读页正文节奏优先于视觉紧凑；
- 我的回顾可以有轻微仪式感，但不堆叠装饰。

## 6. 容器与布局

### 6.1 页面宽度策略

| 页面类型 | 宽度策略 |
| --- | --- |
| 品牌说明页 | 允许更宽 section 容器 |
| 公共策展页 | 主内容区稳定，便于扫描 |
| 个人工作台页 | 主内容优先，辅助区弱化 |
| 阅读详情页 | 正文宽度优先阅读舒适度 |
| 设置配置页 | 表单路径与说明关系优先 |

### 6.2 侧栏策略

**适用于内容工作台型页面：**

- 左侧：导航 / 入口；
- 中间：主内容区；
- 右侧：辅助信息区。

**执行要求：**

- 主内容区永远是视觉中心；
- 右侧辅助区对比度低于主内容区；
- 辅助区不承载主决策信息；
- 中小屏不强制保留三栏；
- 右侧区不应堆叠营销 CTA。

### 6.3 正文宽度

阅读页正文需保持可长时间阅读的宽度，不追求多列塞满。

**执行要求：**

- 正文最大宽度应控制在舒适阅读范围；
- 若提供宽窄模式，需保持阅读逻辑不变；
- AI 伴读侧栏不应压缩正文到难读；
- 双语对照时需要额外考虑行长和段落对应关系。

### 6.4 移动端布局

移动端不是桌面端缩小版。

**执行要求：**

- 主信息优先；
- 操作收敛；
- 筛选器轻量；
- Bottom Sheet 优先承载辅助配置；
- 底部导航和浮层不遮挡核心内容；
- 不依赖 hover；
- 卡片信息可压缩，但不能丢失判断所需的标题、来源、摘要或时长等关键信息。

## 7. 页面模板执行规范

本章承接 5-DESIGN.md 的页面意图模型。

### 7.1 品牌说明页

**适用场景：**

- 首页；
- About；
- Pricing；
- 功能介绍页；
- 产品发布页；
- 活动页。

**页面目标：**

- 快速说明 BestBlogs 是什么；
- 建立品牌气质；
- 展示核心价值；
- 引导用户开始阅读或了解 Pro。

**执行要点：**

- H1 + 副标题 + 关键路径 CTA 构成首屏；
- 主标题优先使用「发现真正适合你的高质量内容」；
- 视觉强调私人阅读助手和高质量信息输入；
- 不堆砌功能点；
- 不使用炫技式 AI 视觉；
- Pricing 页面强调完整私人阅读助手工作流，而不是 VIP 身份；
- 插图与视觉辅助服从信息表达。

**不应出现：**

- Dashboard 式大面板；
- 复杂功能矩阵；
- 过强 AI 发光效果；
- 倒计时焦虑；
- 海量内容话术。

### 7.2 公共策展页

**适用场景：**

- 每日早报；
- 精选周刊；
- 主题解读；
- 内容广场。

**页面目标：**

- 展示 BestBlogs 的公共判断力；
- 让匿名用户也能完成内容判断；
- 建立信任；
- 引导用户从公共内容进入个人阅读流。

**执行要点：**

- 公共质量信号清楚但不过量；
- 来源透明；
- 标题和摘要服务快速判断；
- 不引入黑盒个性化默认排序；
- 转化提示应轻量、上下文相关；
- 不用登录墙阻断公共内容阅读；
- 内容广场默认使用「精选 / 最新」这类公共视角。

**不应出现：**

- 个性化信号伪装成公共判断；
- 热度榜替代质量判断；
- 首屏强制升级弹窗；
- 信息流式无限沉迷设计；
- 标题党视觉。

### 7.3 个人工作台页

**适用场景：**

- 我的早报；
- 我的关注；
- 我的阅读；
- 我的回顾。

**页面目标：**

- 让用户感受到这里是自己的阅读空间；
- 展示和用户相关的内容；
- 帮助用户形成早晚阅读节奏；
- 让关注、阅读、收藏、划线和回顾形成连续关系。

**执行要点：**

- 个性化信号必须可解释；
- 相关性信号轻量但清楚；
- 用户已关注来源和主题应可见；
- 我的早报有编辑递交感；
- 我的回顾比早报更内省；
- 我的阅读强调回看和复用；
- 我的关注强调用户主动选择信息边界。

**不应出现：**

- 不可解释的推荐堆叠；
- 与用户无关的公共广告式表达；
- 过多配置项；
- 强打断 Pro 引导；
- 普通瀑布流式消费体验。

### 7.4 阅读详情页

**适用场景：**

- 文章详情；
- 播客详情；
- 视频详情；
- 推文 / 线程详情；
- 早报阅读页；
- 主题解读详情。

**页面目标：**

- 帮助用户快速理解内容；
- 支持深读；
- 让 AI 伴读和翻译自然辅助阅读；
- 支持收藏、划线、稍后阅读等轻量动作。

**执行要点：**

- 顶部区域快速说明主题、来源、价值线索；
- 正文区最大化；
- AI 伴读在结构上从属于阅读区；
- 相关推荐放在次级位置；
- 划线、收藏、提问等动作轻量存在；
- Pro 引导仅允许出现在自然断点；
- 匿名访客转化 CTA 放在正文之后等自然位置，不阻断阅读。

**不应出现：**

- 正文旁过重操作区；
- 频繁闪动或吸附的浮层；
- 与正文竞争的视觉噪音；
- 连续阅读区插入打断式升级弹层；
- 通用聊天入口。

### 7.5 设置与配置页

**适用场景：**

- 我的关注设置；
- 来源管理；
- 兴趣设置；
- Domain 篇数配置；
- Pro 设置；
- 账号与偏好设置；
- 通知和邮件偏好。

**页面目标：**

- 帮助用户完成明确配置；
- 让用户理解配置结果；
- 降低出错概率；
- 提供控制感，但不制造复杂调权负担。

**执行要点：**

- 信息分组明确；
- 每组只解决一个问题；
- 配置项后应说明结果；
- 表单控件、说明文字、保存反馈形成稳定结构；
- 用户通过关注、阅读、收藏、划线和反馈表达偏好；
- 不开放复杂手动调权。

**不应出现：**

- 技术术语堆叠；
- 不可解释的系统状态；
- 过大的装饰模块；
- 权重仪表盘式配置；
- 多层嵌套表单。

## 8. 内容对象 UI 规范

### 8.1 通用内容卡片

**推荐结构：**

1. 标题；
2. 摘要；
3. 元信息；
4. 标签区；
5. 信号区；
6. 轻量操作区。

**统一原则：**

- 标题优先于封面；
- 摘要不宜过长；
- 元信息和标签保持弱化；
- 操作区默认不抢眼；
- 同卡片最多一个强信号；
- 移动端不能丢失核心判断信息。

### 8.2 文章卡片

**重点字段：**

- 标题；
- 摘要；
- 来源；
- 日期；
- 阅读时长；
- 标签。

**执行要求：**

- 封面图可有可无；
- 无图时不补无意义占位图；
- 阅读时长清晰但不抢标题注意力；
- 翻译状态可以轻量呈现；
- 原文入口不应比标题更抢眼。

### 8.3 播客卡片

**重点字段：**

- 标题；
- 节目 / 嘉宾；
- 时长；
- 摘要；
- 是否有转录；
- 来源。

**执行要求：**

- 时长信息必须明显；
- 摘要比文章卡片更偏值得听什么；
- 若有转录标识，可作为辅助判断信号；
- 不把播放按钮做成唯一主视觉；
- 长播客应突出章节或重点摘要。

### 8.4 视频卡片

**重点字段：**

- 标题；
- 时长；
- 摘要 / 章节；
- 来源；
- 转录状态。

**执行要求：**

- 不让大缩略图主导整个卡片；
- 章节存在时优先展示价值；
- 时长与标题组合帮助判断阅读成本；
- 长视频深度伴读入口不应出现在普通列表强主位；
- 封面图服务识别，不服务点击刺激。

### 8.5 推文 / 线程卡片

**重点字段：**

- 作者；
- 文本片段；
- 线程上下文；
- 价值说明；
- 来源平台。

**执行要求：**

- 优先头像 + 文本，不必强求缩略图；
- 更强调是否值得展开；
- 弱化社交平台式互动元素；
- 不展示点赞、转发等容易引向热度判断的指标；
- 作者可信度和上下文比平台互动更重要。

### 8.6 早报条目

**重点字段：**

- 条目标题；
- 条目摘要；
- 来源；
- 推荐理由；
- 快速动作。

**执行要求：**

- 像可信赖编辑递交内容；
- 条目密度较高但层次清晰；
- 单条操作简洁，不做重卡片化堆叠；
- 每条都应帮助用户判断是否值得进一步阅读；
- 公共早报突出公共判断；
- 我的早报突出个人相关性和推荐解释。

### 8.7 主题解读卡片 / 列表项

**重点字段：**

- 标题；
- 类型：事件 / 领域 / 人物组织 / 对比；
- 结构摘要；
- 引用来源数量；
- 最近更新；
- 延伸阅读入口。

**执行要求：**

- 强调结构化理解，而不是普通文章摘要；
- 可以展示目录感或关键小节；
- 不做成热点专题页；
- 视觉上比普通文章更有层级，但不喧宾夺主；
- 引用和延伸阅读是可信度信号，不是装饰。

### 8.8 我的回顾区块

**重点字段：**

- 日期；
- 今日阅读数量；
- 主要主题；
- 值得回看的内容；
- 稍后阅读；
- 今日一问；
- 阅读足迹。

**执行要求：**

- 语气克制、内省；
- 不做成生活方式打卡页；
- 不使用装饰性 emoji；
- 不使用感叹号；
- 强调结构和回看，而不是情绪陪伴；
- 与我的阅读中的历史、收藏、划线、笔记建立关系；
- 不把回顾做成另一个普通信息流。

### 8.9 关注 / 订阅源卡片

**重点字段：**

- 来源名称；
- 来源类型；
- 描述；
- 语言；
- 更新频率；
- 最近内容；
- 关注状态；
- 来源身份：官方源 / 公共源 / 社区源。

**执行要求：**

- 关注动作清楚；
- 取消关注可达；
- 来源身份清晰；
- 不把来源卡片设计成广告卡片；
- 来源描述要帮助判断是否值得长期关注；
- 关注行为要让用户感受到自己在塑造个人阅读流；
- 「关注」用于来源动作，「订阅」只用于 Pro 付费关系。

## 9. 体验状态执行规范

本章承接 5-DESIGN.md 的阅读体验五态。

### 9.1 扫描态

**适用场景：**

- 内容广场；
- 每日早报；
- 精选周刊；
- 我的早报；
- 我的关注；
- 列表页；
- 搜索结果。

**目标：**

- 尽快判断；
- 尽量少打扰；
- 让用户愿意进入下一步。

**UI 执行规则：**

- 卡片标题优先；
- 摘要短而有信息量；
- 元信息单行优先；
- 标签最多 3 个；
- 操作弱化；
- 不依赖 hover；
- 不展示过多 badge；
- 不让缩略图压过标题；
- 移动端保留标题、来源和关键信号。

### 9.2 理解态

**适用场景：**

- 刚进入详情页；
- 摘要区；
- 章节区；
- AI 伴读初始区；
- 翻译入口附近。

**目标：**

- 帮用户迅速看懂核心价值；
- 帮用户做出继续读、暂时离开、收藏备用的判断。

**UI 执行规则：**

- 顶部摘要清楚；
- 关键点结构化；
- 章节入口可扫描；
- AI 伴读入口明确但弱于正文；
- 翻译入口与原文关系清楚；
- 不把理解态做成聊天页；
- 不展示过多 AI 能力按钮。

### 9.3 深读态

**适用场景：**

- 正文连续阅读；
- 播客 / 视频长时间收听或观看；
- 划线和笔记；
- 翻译对照阅读。

**目标：**

- 安静；
- 稳定；
- 少打断。

**UI 执行规则：**

- 正文宽度控制；
- 浮动工具弱化；
- 推荐区放在次级位置；
- 不插入打断式 Pro CTA；
- 减少动画；
- 减少 badge；
- 不自动跳下一篇；
- AI 伴读是旁侧助手，不是中央主角。

### 9.4 回顾态

**适用场景：**

- 我的回顾；
- 我的阅读；
- 历史；
- 收藏；
- 划线；
- 笔记；
- 稍后阅读。

**目标：**

- 帮助用户重新看到已读内容中的结构与重点；
- 让阅读结果可回顾、可积累、可复用；
- 形成早晚节奏。

**UI 执行规则：**

- 日期、主题、已读内容、稍后阅读清楚；
- 不做普通信息流；
- 不使用过度情绪化陪伴文案；
- 不使用装饰性 emoji；
- 结构优先于气氛；
- 回顾内容和原始内容之间可跳转；
- 回顾态只出现在我的空间，不出现在公共策展层。

### 9.5 选择态

**适用场景：**

- 新用户引导；
- 我的关注；
- 来源管理；
- 兴趣设置；
- Domain 篇数配置；
- 关注 / 取消关注；
- 不感兴趣反馈。

**目标：**

- 帮助用户主动建立自己的信息输入边界；
- 让用户知道自己在塑造个人阅读流；
- 降低配置成本；
- 提供控制感，但不要求复杂调权。

**UI 执行规则：**

- 选择项要可理解；
- 候选建议不能像强制推荐；
- 关注和取消关注清楚；
- 配置结果要有反馈；
- 不把兴趣设置做成复杂仪表盘；
- 不开放手动权重调节；
- 空状态引导用户建立信息输入边界；
- 不把选择态做成问卷负担。

## 10. 信号系统执行规范

**BestBlogs 中必须区分三类信号：**

1. 质量信号；
2. 相关性信号；
3. 个性化信号。

这三类信号不能混为一谈。

### 10.1 质量信号

**回答：**

这条内容本身值不值得看？

**例如：**

- 精选；
- 编辑推荐；
- 高质量来源；
- 深度内容；
- 公共早报入选；
- 主题解读引用。

**主映射：**

- featured / quality badge；
- 卡片左侧弱强调线；
- 详情页「为何精选」简短说明；
- 公共入口中的编辑说明。

**规则：**

- 高置信；
- 克制表达；
- 不泛滥；
- 公共策展层优先使用；
- 同卡片最多 1 个强质量信号；
- 不使用热度指标伪装质量。

### 10.2 相关性信号

**回答：**

它和我当前关注的方向是否相关？

**例如：**

- 与你关注的主题相关；
- 与你最近阅读相关；
- 属于你常读来源类型；
- 与你关注的 Domain 相关。

**主映射：**

- 轻量说明文案；
- 弱化标签；
- 推荐原因中的次级解释；
- tooltip 或折叠说明。

**规则：**

- 视觉权重低于质量信号；
- 不伪装成公共权威判断；
- 主要出现在我的空间；
- 不应在公共策展层出现用户相关性表达。

### 10.3 个性化信号

**回答：**

为什么它此刻出现在我面前？

**例如：**

- 来自你关注的来源；
- 加入你的早报；
- 与你最近阅读相关；
- 因你的 Domain 篇数设置被优先展示；
- 你收藏过相似内容；
- 你最近减少了某类内容的阅读。

**主映射：**

- RecommendationReason；
- 轻量「为什么推这条」；
- 来源于入口语义，例如我的早报、我的关注；
- 个性化说明区。

**规则：**

- 仅出现在我的空间；
- 不出现在公共策展层；
- 不伪装成公共权威判断；
- 不暴露权重调节入口；
- 不展示模型内部参数；
- 要尽量让用户理解其来源逻辑。

### 10.4 信号组合规则

| 组合 | 规则 |
| --- | --- |
| 质量 + 相关性 | 质量信号更强，相关性信号更轻 |
| 质量 + 个性化 | 可用于我的空间，但需说明一个是内容质量，一个是为什么适合我 |
| 相关性 + 个性化 | 可合并进推荐原因 |
| 三者同时出现 | 默认折叠，只保留一个主表达 |

**视觉权重建议：**

质量信号 > 个性化信号 > 相关性信号

**禁止：**

- 用「平台认证必读」伪装个性化；
- 用「因为你看过」伪装公共质量判断；
- 公共策展层出现「为你」类信号；
- 同卡片出现多个强 badge；
- 把热度、阅读量、点击量作为质量主信号。

## 11. 组件规范

### 11.1 Button

**层级：**

- Primary：当前场景唯一主动作；
- Secondary：常见但非主线动作；
- Ghost / Tertiary：辅助动作；
- Destructive：破坏性动作。

**规则：**

- 一个区域尽量只有一个 Primary；
- 破坏性动作不用 Amber；
- 文案使用结果导向表达；
- Primary 不用于普通筛选和次级跳转；
- Loading 状态应避免重复提交。

### 11.2 Icon Button

**适用：**

- 收藏；
- 分享；
- 稍后读；
- 标记已读；
- 更多操作；
- 展开 / 收起。

**规则：**

- 必须有 aria-label；
- 不单独承担关键主路径；
- 小尺寸时确保可触控面积满足可访问性要求；
- 图标语义要清楚，不用装饰性 emoji 替代功能图标。

### 11.3 Badge / Tag

**variant 建议：**

- category
- language
- topic
- status
- pro
- featured
- source
- personalized

**规则：**

- 单行不超过 3 个主要标签；
- Featured / Pro / 状态类需明显区分；
- 不能形成 badge 墙；
- 标签不替代摘要；
- 状态 badge 不应作为唯一信息来源。

### 11.4 Tabs

**适用：**

- 内容入口切换；
- 阅读模式切换；
- 设置分组切换；
- 我的阅读 collection 切换。

**规则：**

- tab 数量不宜过多；
- 选中态清晰；
- 移动端可横向滚动；
- 不用 Tabs 承载过深层级；
- Tabs 切换不应造成用户上下文丢失。

### 11.5 Search

**适用：**

- 内容搜索；
- 来源搜索；
- 设置项搜索；
- 我的阅读搜索。

**规则：**

- 输入框优先简洁；
- 占位提示使用用户语言；
- 不叠加过多高级筛选入口；
- 搜索结果应保留来源、类型和匹配原因；
- 空结果应提供下一步建议。

### 11.6 Filters

**适用：**

- 列表页筛选；
- 内容广场；
- 我的关注；
- 我的阅读；
- 搜索结果。

**规则：**

- 默认轻量显示；
- 常用项先展示；
- 高级项折叠；
- 移动端优先使用横向 pills、底部抽屉或轻量弹层；
- 筛选项不应比内容更抢眼；
- 筛选结果变化需要可感知。

### 11.7 Dropdown / More Menu

**适用：**

- 辅助操作；
- 管理型动作；
- 次级导航。

**规则：**

- 重要操作不应藏在过深层级；
- 菜单项命名使用统一术语；
- 危险操作分组并明显区隔；
- 移动端确保触控区域充足；
- More Menu 不应成为功能垃圾桶。

### 11.8 Dialog / Sheet

**适用：**

- 确认操作；
- 快速配置；
- 移动端补充面板；
- Pro 能力说明；
- 来源管理轻量操作。

**规则：**

- 对话框只解决一个问题；
- 避免在对话框里再堆多层流程；
- 移动端可用 bottom sheet 承载轻量配置；
- 不能用于首屏强制升级；
- 不能用于打断正文阅读；
- 可关闭性明确。

**反骚扰反例：**

- ❌ 首屏 Modal 强制升级；
- ❌ 倒计时；
- ❌ 不可关闭遮罩；
- ❌ 阻断阅读主流程的「立即开通 Pro 才能继续」；
- ✅ 允许：能力天花板触达时的轻量说明。

### 11.9 Toast / Feedback

**适用：**

- 保存成功；
- 标记完成；
- 错误反馈；
- 撤销入口；
- 配额提示；
- 状态更新。

**规则：**

- 信息短、明确；
- 成功反馈不过度抢眼；
- 错误反馈提供可理解原因或下一步；
- 支持撤销的动作应提供撤销入口；
- 不把 Toast 当营销渠道。

**反例：**

- ❌ 你今天竟然读了 N 篇；
- ❌ 错过了重要内容；
- ❌ AI 已替你读完；
- ❌ 无需打开即可掌握；
- ❌ 非用户行为触发的升级 Pro Toast；
- ✅ 已保存；
- ✅ 已加入稍后读；
- ✅ 翻译额度已使用 8/10。

## 12. BestBlogs 专属组件规范

### 12.1 ContentCard

承载内容扫描和判断。

**必须支持：**

- 标题；
- 摘要；
- 来源；
- 时间 / 时长；
- 内容类型；
- 质量 / 相关性 / 个性化信号；
- 轻量操作。

**规则：**

- 标题是主视觉；
- 封面图从属于内容；
- 同卡片最多一个强信号；
- 卡片可点击区域清晰；
- 操作区不抢标题；
- 移动端信息压缩但不丢失核心判断。

### 12.2 SourceCard

承载关注 / 订阅源判断。

**必须支持：**

- 来源名称；
- 来源描述；
- 来源类型；
- 语言；
- 更新频率；
- 来源身份；
- 关注状态；
- 最近内容。

**规则：**

- 关注按钮清楚；
- 取消关注可达；
- 官方源 / 公共源 / 社区源身份明确；
- 不像广告卡片；
- 来源描述服务长期关注判断；
- 不使用订阅作为来源动作。

### 12.3 RecommendationReason

承载个性化解释。

**适用：**

- 我的早报；
- 我的关注；
- 内容详情页；
- 推荐列表；
- AI 伴读上下文说明。

**规则：**

- 文案短；
- 解释用户能理解；
- 不暴露模型参数；
- 不展示权重；
- 不伪装成公共质量判断；
- 可折叠；
- 不能强行给每条内容堆复杂解释。

**推荐文案方向：**

- 来自你关注的来源；
- 与你最近阅读相关；
- 属于你关注的主题；
- 加入你的今日早报；
- 因你的 Domain 设置被优先展示。

### 12.4 TranslateToggle

- 译文；
- 双语对照；
- 翻译中；

**规则：**

- 原文与译文关系清楚；

- 翻译失败不破坏阅读；

### 12.5 CopilotPanel

承载 AI 伴读。

**适用：**

- 内容详情；
- 长视频 / 长播客；
- 主题解读；
- 我的阅读；
- 回顾态辅助。

**规则：**

- 从属于阅读区；
- 默认不抢正文主角；
- 初始问题围绕当前内容；
- 不使用「问我任何事」；
- 不变成通用聊天入口；
- 输出尽量可追溯到内容；
- Pro 边界使用 inline gate 或自然断点说明。

### 12.6 ProInlineGate

承载 Pro 引导。

**适用：**

- 自定义视图；
- Domain 篇数配置；
- 我的早报；
- 我的回顾。

**规则：**

- 优先引导试用；
- 不使用强弹窗；
- 不使用倒计时；
- 不使用 VIP / 尊享；
- 说明能力边界；
- 只在上下文相关位置出现；
- CTA 文案结果导向。

**推荐文案：**

- 试用完整的私人阅读助手；
- 解锁完整的我的早报；
- 使用更完整的 AI 伴读；
- 获得更高配额的跨语言阅读。

### 12.7 ReadingToolbar

承载阅读中的轻量操作。

**适用：**

- 收藏；
- 稍后读；
- 划线；
- 分享；
- 翻译；
- AI 伴读；

**规则：**

- 深读态弱化；
- 移动端不遮挡正文；
- 操作数量克制；

### 12.8 EmptyState

承载空状态引导。

**必须回答：**

## 13. 状态规范

### 13.1 基础交互状态

**所有交互组件至少定义：**

- default；
- hover；
- focus-visible；
- active；
- selected；
- disabled；
- loading。

**要求：**

- focus-visible 明确；
- disabled 不只靠低透明度；
- loading 状态防止重复提交；
- selected 和 active 不混淆。

### 13.2 内容状态

**统一支持：**

- 未读；
- 已读；
- 稍后读；
- 已收藏；
- 已划线；
- 不感兴趣；
- 已分享。

术语与命名应使用 10-TERMINOLOGY.md 中已确认的用户可见表达。

### 13.3 已读状态

**推荐组合表达：**

- 标题对比度略降；
- 摘要权重降低；
- 操作按钮更隐性；
- 可选轻量「已读」提示。

不能只靠颜色细微变化。

### 13.4 稍后读 vs 收藏

**必须区分：**

| 状态 | 语义 |
| --- | --- |
| 稍后读 | 短期任务感 |
| 收藏 | 长期保存感 |

**要求：**

- 图标和文案应区分；
- 不都叫保存；
- 稍后读适合进入回顾和我的阅读；
- 收藏适合长期沉淀。

### 13.5 不感兴趣

**表现方向：**

- 减少类似内容；
- 让系统更懂我；
- 可撤销；
- 不做强烈负面反馈。

**避免表现成：**

- 强烈删除；
- 永久屏蔽世界；
- 惩罚来源；
- 过度负面情绪。

### 13.6 页面状态

**所有页面都应设计：**

- loading / skeleton；
- empty；
- error；
- no permission.
- quota exceeded.
- Pro required.
- onboarding needed.
- offline / network error.

### 13.7 空状态

**空状态至少要回答：**

1. 为什么这里为空；
2. 可以做什么；
3. 这样做之后会得到什么。

**不同场景要求：**

| 场景 | 空状态方向 |
| --- | --- |
| 我的关注 | 引导关注来源 |
| 我的早报 | 引导试用 / 完善关注 |
| 我的阅读 | 引导阅读、收藏、划线 |
| 我的回顾 | 克制说明今天暂无阅读记录 |
| 搜索 | 建议换关键词或减少筛选 |
| 内容广场 | 说明当前筛选无结果 |

### 13.8 权限状态

**权限状态包括：**

- 未登录；
- Free 用户访问 Pro 能力；
- Pro 已过期；
- 配额耗尽；
- 私有内容无权限；
- OpenAPI quota exceeded。

**执行规则：**

- 说明原因；
- 给出下一步；
- 不责备用户；
- 不使用焦虑文案；
- 不把权限状态做成错误状态；
- Pro 权限状态优先引导试用或升级完整工作流。

### 13.9 配额状态

配额状态用于 AI 伴读、翻译、来源数量等能力。

**执行规则：**

- 使用用户可理解的动作次数；
- 不暴露 token / 字符数；
- 接近上限时可轻量提示；
- 到达上限时 inline 说明；
- 提供升级或明日恢复说明；
- 不使用强弹窗。

**接近上限的阈值表达（关注源 / 阅读流类配额）：**

- 使用率 <80%：保持 muted 文字（`text-muted-foreground`），不渲染进度条；
- 使用率 ≥80%：文字变 amber（`text-amber-700 dark:text-amber-300`），渲染细 amber 进度条（`bg-amber-600`，width = 实际百分比）；
- 到达 100%：参考 §10「Pill / Tag / Badge」warning 条款追加 inline 警示；
- 阈值常量在前端代码内（如 `QUOTA_AMBER_THRESHOLD = 0.8`），无需 ConfigKey 远程开关。

**示例：**

- 今日翻译额度已使用 8/10。
- 今日 AI 伴读次数已用完，明天会自动恢复。
- 关注源 480 / 500（已 ≥80%，amber + 进度条出现）。
- Pro 可获得更高配额的跨语言阅读能力。

## 14. AI 伴读与翻译 UI 规范

### 14.1 AI 伴读入口

**入口策略：**

| 场景 | 策略 |
| --- | --- |
| 列表页 | 轻入口，不占主视线 |
| 详情页理解态 | 清楚可见，帮助理解当前内容 |
| 深读态 | 从属于正文，按需出现 |
| 长视频 / 长播客 | 可更主动提供章节、摘要和提问 |
| 回顾态 | 可帮助整理线索，但不情绪化 |
| 主题解读 | 可辅助解释结构和延伸问题 |

**禁止：**

- 问我任何事；
- 通用聊天入口；
- 大面积 AI 发光效果；
- AI 面板默认覆盖正文；
- AI 输出替代原文；
- AI 结论不可追溯。

### 14.2 AI 伴读面板

**结构建议：**

1. 当前内容上下文；
2. 推荐问题；
3. 摘要 / 章节 / 关键点；
4. 用户提问；
5. 引用或来源回指；
6. 配额 / Pro 状态提示。

**执行规则：**

- 不默认占满页面；
- 移动端可用 Sheet；
- 桌面端可用侧栏；
- 面板打开后正文仍可阅读；
- AI 回答要尽量短、结构清晰；
- 错误状态要解释原因和下一步。

### 14.3 沉浸式翻译入口

**入口位置：**

- 内容详情顶部；
- 正文工具条；
- 段落级辅助入口；
- AI 伴读相关动作区。

**执行规则：**

- 原文与译文关系清楚；
- 翻译状态可感知；
- 支持切回原文；
- 双语对照不破坏段落节奏；
- 翻译配额提示 inline 展示；
- 翻译失败时保留原文阅读。

### 14.4 原文 / 译文 / AI 解释关系

**优先级：**

原文 > 译文 > AI 解释

**执行规则：**

- 原文始终可达；
- 译文不能伪装成原文；
- AI 解释不能替代正文；
- 术语解释应贴近上下文；
- 复杂解释不应默认展开；
- 用户应能在原文、译文和解释之间自然切换。

## 16. 动效规范

### 16.1 动效原则

动效服务理解，而不是装饰。

**执行要求：**

- 动效只服务状态切换、结构变化、轻微反馈；
- 不使用长时间循环动画；
- 不使用无意义背景动效；
- 不让动效与阅读主区竞争注意力；
- 尊重 prefers-reduced-motion。

### 16.2 允许的动效场景

- hover 反馈；
- 展开 / 折叠；
- 轻量卡片入场；
- 选中态切换；
- 对话框 / 抽屉开合；
- loading 到内容的过渡；
- 翻译结果出现；
- AI 回答流式输出。

### 16.3 约束

- 禁止大面积无目的动画；
- opacity-0 初始态必须配合 animation-fill-mode: forwards；
- 所有入场动画必须可被关闭或自然跳过；
- 列表加载不使用炫目骨架动画；
- 深读页尽量减少持续动画元素；
- 不使用 AI 魔法感动画。

## 17. 响应式与可访问性

### 17.1 响应式核心原则

- 移动端优先快速扫读和轻量操作；
- 桌面端优先长时间阅读和多维筛选；
- 不允许关键判断信息只在 hover 可见；
- 不做桌面布局等比缩放。

### 17.2 移动端

**执行要求：**

- 最小触控区域 44×44；
- 支持安全区；
- 筛选器适合横向滚动或轻量入口；
- 列表项可减少摘要长度，但不能省掉核心判断信息；
- 操作收敛到主线和更多菜单；
- 底部导航与浮层不能遮挡核心内容；
- AI 伴读优先 Sheet 或独立层；
- 深读态工具条弱化。

#### 视口高度单位

移动端布局必须使用 `100dvh`，而非 `100vh`，以正确处理 iOS Safari / Android Chrome 地址栏动态显示 / 隐藏导致的视口尺寸变化。

`min-h-screen` 等价于 `100vh`，在移动端可能导致底部内容被工具栏遮挡，应替换为：

```tsx
className="min-h-[100dvh]"
```

**受影响组件包括但不限于：**

- 全屏页面；
- 侧边栏；
- 详情页容器；
- 登录页；
- 引导页；
- 移动端 Sheet。

### 17.3 桌面端

**执行要求：**

- 允许更完整的摘要与辅助信息；
- 允许右侧辅助区，但不与主阅读区竞争；
- 筛选维度可适度增加，但保持轻量默认态；
- 详情页正文宽度优先；
- 多栏布局应保持主内容清晰。

### 17.4 Hover 替代策略

凡是在桌面依赖 hover 才暴露的信息，移动端都必须提供可见或可触达替代方案。

**不能只在 hover 中出现：**

- 推荐原因；
- 主要操作；
- 配额状态；
- Pro 边界；
- 内容来源；
- 翻译入口。

### 17.5 可访问性基础要求

**最小验收阈值：**

- 正文与关键 UI 文本满足 WCAG AA 对比度；
- 普通文本对比度 >= 4.5:1；
- 大文本对比度 >= 3:1；
- 所有可交互控件必须可通过键盘到达并触发；
- focus-visible 在浅色和暗色下均清晰可见；
- 非文本状态必须同时有文本或图标语义，不可仅靠颜色；
- Icon Button 必须有 aria-label；
- Dialog / Menu / Tabs 使用标准语义角色。

### 17.6 内容对象语义

**建议：**

- 列表页内容对象使用语义化结构；
- 卡片可使用 role="article"；
- 标题结构遵循文档层级，不随意跳级；
- tab、dialog、menu 等使用标准语义角色；
- 状态类 badge 不能作为唯一信息来源。

### 17.7 状态可感知

- 已读 / 未读不能只靠颜色区分；
- 成功 / 错误反馈应同时具备文字提示；
- loading 状态应可被读屏理解；
- AI 生成中状态要说明当前动作；
- 翻译失败要给出可理解说明；
- 配额耗尽不是 error，应作为权限 / 配额状态处理。

- 是否有移动端方案；
- 是否满足可访问性要求。

### 18.3 Pro / AI / 信号专项 Checklist

**如果页面涉及 Pro、AI 或推荐信号，额外检查：**

- Pro 引导是否试用优先；
- Pro 引导是否避免强弹窗；
- 配额提示是否 inline；
- AI 是否从属于阅读；
- AI 是否避免「问我任何事」；
- 推荐原因是否用户能理解；
- 是否没有暴露模型权重；
- 公共策展层是否没有个性化信号；
- 我的空间是否没有伪装公共质量判断。

### 18.4 设计决策记录

- 执行层的局部约束更新，优先写入本文件；
- 若影响顶层体验原则，同步更新 5-DESIGN.md；
- 若影响产品定位或跨模块边界，同步更新 2-PRODUCT.md 或 4-ARCHITECTURE.md；
- 若影响品牌表达，同步更新 3-BRAND.md；
- 若影响术语，同步更新 10-TERMINOLOGY.md；
- DECISIONS.md 仅作为历史归档，不作为新增决策入口。

## 19. 一句话总结

6-UI-SPEC.md 的目标，不是让界面元素越来越多，而是让 BestBlogs 的设计原则被稳定、可复用、可验证地执行出来。

**它要确保每一个真实界面都服务于同一个体验目标：**

让用户更快判断什么值得读，更顺畅地理解内容，更安静地进入阅读，并在持续使用中感到 BestBlogs 越来越适合自己。
